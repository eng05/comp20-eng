Security Assessment of Sylvie Abookire’s Marauder’s Map

Prepared by Elwin Ng on 4/20/2015

Introduction

This assessment tests the security of Sylvie's Maruader's Map and the server-side applicationfor it. The Marauder’s Map application retrieves the user’s current location and sends the current location to a database that was also created by Sylvie for assignment 3. The map then displays the locations of people in the class with a marker indicating where they are and how far away they are from the user. The server-side application uses Heroku, Node.js, and the Express web framework. The user’s login, latitude, and longitude are collected and stored inside MongoDB along with the data obtained from other users. Upon request, it displays a list of the check-ins for all logins sorted in descending order by timestamp. I was hired to find security issues within these applications and propose solutions to solve these security issues. In my assessment, I have found a total of 3 security issues.

Methodology

I began this assessment with black box testing, where I tried to hack into or change the data in Marauder’s Map without viewing the source code. This involved using Tamper Data within the browser and curl within the terminal. I then reviewed the source code in order to better understand the security flaws that allowed my hacks to work during black box testing.

Abstract of Findings

Most security issues found within the application were due to failure to validate user input. Users were able to submit scripts and invalid latitudes and longitudes, which could lead to arbitrary and false content being displayed. Users were also able to access private information through an injection attack, which could be disastrous. To resolve these security issues, I would suggest implementing further validation checks of user input.

Issues Found

1. Issue: Cross Site Scripting (XSS)

  • Location: /sendLocation
  • Severity of issue: High, hackers can insert unexpected input, execute arbitrary commands and display arbitrary content in a victim user's browser
  • Description of issue: Inserting javascript using the script tag would cause the user’s application to run the code and present all users with fraudulent web content. I tested this by inserting <script>alert('HACKED BY ELWIN');</script> into the login field using curl. Now, every time the page is accessed, an alert with the message “HACKED BY ELWIN” is displayed.
  • Proof of vulnerability:
  • Resolution: Remove the ability for data to be interpreted as code by paying attention to the angle brackets. When storing user input, change < to < and change > to >, which are their html equivalants. By doing so, the application will no longer interpret the script tags as executable.

2. Issue: Falsification of Data

  • Location: /sendLocation
  • Severity of issue: Medium. Even though it is quite easy for hackers to submit false data, the program is still able to run normally.
  • Description of issue: The user can provide false data that is inaccurate by using Tamper Data or curl. For example, the user can input a latitude and longitude of 999, which clearly does not exist. Although Marauder’s Map simply does not show this user’s marker, it should definitely show an error message if the latitude and longitude are invalid numbers and make sure that the user’s actual location is stored.
  • Proof of vulnerability:
  • Resolution: Server-side input validation. The user can include code “if (lat <= 90 && lat >= -90 && lng <= 180 && lng >= -180)” in order to make sure the latitude and longitude numbers are real, valid numbers.

3. Issue: Injection Attack

  • Location: /location.json
  • Severity of issue: High, the user can query the database to access private information without authentication
  • Description of issue: The user can type in “/location.json?login[$ne]=anyrandomstring” in order to return information about all user’s in the system. Although, there is no private information currently stored, this would be a big problem if there were indeed private login information, passwords, or check-in’s that a hacker should not be able to see and accesss
  • Proof of vulnerability:
  • Resolution: Check the post body to see if there are opening and closing brackets in the login input field, such as [$ne]. If there are, ignore the queries containing them and do not save into database. Alternatively, you can also code the brackets to their HTML equivalent.

Conclusion

Issues with Marauder’s Map and its associated server side application make it easy for a hacker to run scripts, enter false information, or gather private information from the database. Implementing my suggested resolutions would definitely help make these applications more secure. Even with these changes, a hacker could still enter valid longitude and latitude values that do not represent their real location. The hacker could pretend to be anywhere in the world. Future considerations include utilizing a second location tracker to ensure that the location is accurate. Yet a sophisticated hacker can most likely always try to find a way around it. Having an accurate location is very important to Marauder’s Map because users expect accuracy when using the product. After making the suggested changes, I would highly recommend conducting a follow-up security analysis, which would cost around $250, and follow up with the latest website security developments and vulnerabilities.
